# AGPA v0.2 Architecture Improvement

## 1. Motivation

Current `agpa-engine.ts` (500 lines) conflates 6 responsibilities: type definitions, YAML parsing, condition evaluation, event management, state persistence, and statistics. Each `poll()` call scans all events linearly (O(n)), and `state.json` writes are not atomic.

This spec refactors the engine into a modular architecture inspired by two popular MCP projects:

- **Exa MCP Server** (TypeScript, 4.4k ⭐) — clean module boundary, `register*Tool()` pattern, config-driven tool enablement
- **Codebase Memory MCP** (C, 2.4k ⭐) — incremental caching over full scans, atomic persistence

## 2. Engine Module Split

**File tree:**

```
src/engine/
├── types.ts           # All types: EventType, Condition, AchievementDefinition, EvaluationResult, etc.
├── yaml-parser.ts     # Minimal YAML → AchievementDefinition[] (pure function, no deps)
├── evaluator.ts       # 6 condition evaluators (pure functions: events → EvaluationResult)
├── counter-cache.ts   # Incremental counter cache (O(1) poll)
├── store.ts           # Atomic state persistence + event log WAL
└── engine.ts          # Orchestrator — thin coordinator (init, track, poll, stats)
```

### 2.1 types.ts

**Move all types from the current monolithic engine into a single file.** No logic changes.

Key exports: `EventType`, `TrackedEvent`, `AchievementDefinition`, `Condition`, `ConditionType`, `RarityLevel`, `EvaluationResult`, `AchievementState`, `AchievementStats`, `EngineOptions`.

### 2.2 yaml-parser.ts

**Extract the `parseYAML()` function as a pure module.**

```typescript
export function parseYAML(text: string): { definitions: AchievementDefinition[] }
```

- Same implementation as current `parseYAML()`
- No engine state needed — pure function, trivially testable
- Throws on malformed YAML (instead of silently ignoring)

### 2.3 evaluator.ts

**Six evaluator functions, each pure and independently testable:**

```typescript
export function evalCounter(events: TrackedEvent[], cond: Condition): EvaluationResult
export function evalThreshold(events: TrackedEvent[], cond: Condition): EvaluationResult
export function evalStreak(events: TrackedEvent[], cond: Condition): EvaluationResult
export function evalSequence(events: TrackedEvent[], cond: Condition): EvaluationResult
export function evalDistinctCount(events: TrackedEvent[], cond: Condition): EvaluationResult
export function evalEvent(events: TrackedEvent[], cond: Condition): EvaluationResult
export function evaluateCondition(cond: Condition, events: TrackedEvent[]): EvaluationResult
```

- `evaluateCondition` is the dispatcher — routes `cond.type` to the right evaluator
- Each evaluator is a pure function: input is `(events, cond)`, output is `EvaluationResult`
- Helpers (`parseWindow`, `matchFilter`, `getField`) stay private to this module
- Unit test each evaluator without loading the engine

### 2.4 counter-cache.ts — Incremental Counter Cache

**The key performance improvement.** Instead of scanning all events on every `poll()`, maintain an incrementally updated counter.

#### Data structure

```typescript
interface CounterCache {
  // event_type → count (for counter conditions)
  eventCounts: Record<string, number>;

  // "eventType:filterHash" → count (for filtered counter conditions)
  filterCounts: Record<string, number>;

  // eventType → sorted list of event dates (for streak conditions)
  streaks: Record<string, string[]>;

  // "eventType:fieldName" → set of values (for distinct_count conditions)
  distinctValues: Record<string, Set<string>>;
}
```

#### API

```typescript
export class CounterCache {
  constructor(snapshot?: SerializedCache);

  /** Call on every track() — O(1) update */
  onTrack(event: TrackedEvent): void;

  /** Get cached count for counter condition */
  getCount(eventType: string, filter?: string): number;

  /** Get streak length for streak condition */
  getStreakLength(eventType: string): number;

  /** Get distinct value count */
  getDistinctCount(eventType: string, field: string): number;

  /** Serialize to JSON for persistence */
  serialize(): SerializedCache;

  /** Bulk rebuild from event log (used on engine init, or when cache is missing) */
  rebuild(events: TrackedEvent[]): void;
}
```

#### Usage

- **On `track()`**: `counterCache.onTrack(event)` — increments relevant counters, updates streak dates, adds to value sets. All O(1).
- **On `poll()`**: condition evaluator reads `counterCache.getCount(type)` instead of scanning events. O(1) per condition.
- **On `init()`**: if serialized cache exists, deserialize. Otherwise call `rebuild(events)` once.

#### Rebuild scenario

On fresh start without cached counters, do a single pass over all events:

```typescript
rebuild(events: TrackedEvent[]) {
  this.eventCounts = {};
  this.filterCounts = {};
  this.streaks = {};
  this.distinctValues = {};
  for (const event of events) {
    this.onTrack(event);
  }
}
```

This is O(n) but happens only once (on init). All subsequent polls are O(number_of_conditions).

#### Consistency guarantee

- Cache is persisted alongside `state.json` as `cache.json`
- If `cache.json` is missing/corrupt, `rebuild()` from event log
- If event log is missing/corrupt, engine falls back to full state reset
- No cross-file consistency: cache is a performance optimization, loss of cache only causes a one-time rebuild

### 2.5 store.ts — Atomic Persistence

**Solves the current JSON-overwrite problem.**

```typescript
export class Store {
  stateDir: string;

  /** Load state from disk */
  load(): { state: AchievementState; events: TrackedEvent[]; cache?: SerializedCache };

  /** Atomic write: write to tmp file, then rename */
  saveState(state: AchievementState): void;
  saveCache(cache: SerializedCache): void;

  /** Append event to log (already atomic on most filesystems) */
  appendEvent(event: TrackedEvent): void;

  /** Full reset */
  reset(): void;
}
```

Atomic write pattern:

```typescript
saveState(state: AchievementState) {
  const tmp = path.join(this.stateDir, 'state.json.tmp');
  const target = path.join(this.stateDir, 'state.json');
  fs.writeFileSync(tmp, JSON.stringify(state));
  fs.renameSync(tmp, target);  // atomic on same filesystem
}
```

### 2.6 engine.ts — Orchestrator

**The public API. Thin coordinator that wires the modules together.**

```typescript
export class AchievementEngine {
  constructor(opts?: EngineOptions);

  init(): this;
  track(eventType: EventType, payload?: EventPayload): TrackedEvent;
  poll(): AchievementDefinition[];
  stats(): AchievementStats;
  resetState(): void;
}
```

Internal flow:

- `init()` → `yaml-parser.parseYAML()` → `store.load()` → `counterCache.rebuild()` if cache missing
- `track()` → generates event → `store.appendEvent()` → `counterCache.onTrack()`
- `poll()` → iterates definitions → `evaluator.evaluateCondition(cond, counterCache)` for each condition → if all met → `store.saveState()`
- `stats()` → computes stats from `state` and `definitions`

## 3. Tool Registration (Exa Pattern)

### Current approach
```typescript
// src/tools/index.ts
export const tools = [trackTool, pollTool, statsTool, showcaseTool, configTool];
```

### New approach

Each tool file exports a `register*Tool(server, engine, config?)` function:

```typescript
// src/tools/track.ts
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import type { AchievementEngine } from '../engine/engine.js';

export function registerTrackTool(server: McpServer, engine: AchievementEngine) {
  server.tool(
    'achievement.track',
    'Record an event for achievement evaluation',
    {
      event_type: z.string(),
      payload: z.object({ ... }).optional(),
    },
    { readOnlyHint: false, destructiveHint: false, idempotentHint: false },
    async ({ event_type, payload }) => {
      try {
        engine.track(event_type, payload ?? {});
        return { content: [{ type: 'text', text: 'ok' }] };
      } catch (e) {
        return { content: [{ type: 'text', text: String(e) }], isError: true };
      }
    },
  );
}
```

Registration center becomes a simple loop:

```typescript
// src/tools/index.ts
export function registerAllTools(server: McpServer, engine: AchievementEngine, config?: ToolConfig) {
  registerTrackTool(server, engine);
  registerPollTool(server, engine);
  registerStatsTool(server, engine);
  if (config?.enableShowcase !== false) registerShowcaseTool(server, engine);
  if (config?.enableConfig !== false) registerConfigTool(server, engine);
}
```

Additional changes per tool:

| Tool | Current behavior | New behavior |
|------|-----------------|--------------|
| `achievement.track` | No error handling | Unified `formatMcpError()` |
| `achievement.poll` | Returns raw list | Pagination via `limit` param, MCP annotations `{readOnlyHint: true, idempotentHint: true}` |
| `achievement.stats` | Returns flat JSON | Rich formatting options via `format` param |
| `achievement.showcase` | 3 actions hardcoded | Configurable slot count |
| `achievement.config` | Single key/value | Batch get/set via `keys`/`values` arrays |

### MCP Annotations per tool

```typescript
track   →  { readOnlyHint: false, destructiveHint: false, idempotentHint: false }
poll    →  { readOnlyHint: true,  destructiveHint: false, idempotentHint: true }
stats   →  { readOnlyHint: true,  destructiveHint: false, idempotentHint: true }
showcase → { readOnlyHint: false, destructiveHint: false, idempotentHint: false }
config  →  { readOnlyHint: false, destructiveHint: false, idempotentHint: false }
```

## 5. Unified Error Handling

```typescript
// src/utils/errors.ts
export interface McpErrorResult {
  content: { type: 'text'; text: string }[];
  isError: true;
}

export function formatMcpError(code: string, message: string): McpErrorResult {
  return {
    content: [{ type: 'text', text: `[${code}] ${message}` }],
    isError: true,
  };
}
```

Error codes:

| Code | Meaning |
|------|---------|
| `ENGINE_NOT_INIT` | Engine not initialized |
| `INVALID_EVENT` | Malformed event payload |
| `UNKNOWN_TOOL` | ool call to unregistered tool |
| `STORE_ERROR` | File I/O failure |
| `PARSE_ERROR` | YAML or cache parse failure |

## 6. Migration Path (from current codebase)

No breaking changes to the public API. The refactor is internal only.

### Step 1 — Module split (no functional change)
1. Create `src/engine/types.ts` — move all type definitions
2. Create `src/engine/yaml-parser.ts` — extract `parseYAML()`
3. Create `src/engine/evaluator.ts` — extract 6 evaluators
4. Reduce `agpa-engine.ts` → `src/engine/engine.ts` — keep only orchestrator logic
5. Update all imports

**Tests**: existing evaluations must produce identical results.

### Step 2 — Counter cache (performance)
1. Create `src/engine/counter-cache.ts`
2. Wire into `engine.track()` and `engine.poll()`
3. Add persistence alongside `state.json`

**Tests**: poll results with cache must match poll results without cache for the same event log.

### Step 3 — Atomic store
1. Create `src/engine/store.ts`
2. Replace inline `readFileSync`/`writeFileSync` calls

**Tests**: simulate crash during write, verify state file is always consistent.

### Step 4 — Tool registration refactor
1. Convert each `src/tools/*.ts` to use `register*Tool()` pattern
2. Update `src/tools/index.ts` to use `registerAllTools()`
3. Add MCP annotations

### Step 5 — Error handler & config (cleanup)
1. Create `src/utils/errors.ts`
2. Add env var support to existing `config.ts`

## 8. Self-Review

- **Placeholders**: None. Every module is accounted for.
- **Internal consistency**: Engine module split matches the dependency graph (types ← yaml-parser, evaluator, store → engine). Counter cache only touches the performance path, not the correctness path.
- **Scope**: Focused on internal architecture. No new features, no UI changes, no protocol changes. Pure refactoring with one performance improvement (counter cache).
- **Ambiguity**: Counter cache rebuild is specified (on missing/corrupt cache, rebuild from event log). Tradeoff between memory (keeping sets for distinct_count) and correctness is explicit — sets are serialized with cache.
