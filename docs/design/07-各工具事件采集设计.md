# 各工具事件采集设计

> 文档目的：逐一分析 5 个 Agent 工具的事件采集能力、实现方案、可靠性评估

---

## 一、概述

### 事件采集三通道

```
通道 A — 原生事件（最可靠）
  Tool 自身机制主动发射事件
  例：CC Hooks、MCP tool.call 自动录制

通道 B — 指令驱动（次可靠）
  Agent 按照 CLAUDE.md / AGENTS.md 指令主动调 achievement.track()
  依赖 Agent 的指令遵守程度

通道 C — 推断/合成（备选）
  没有显式事件的场景，通过扫描 event.log 已有数据推断
  例：task.complete 可能从 tool.call + conversation.message 序列推断
```

### 各工具通道覆盖

```
              通道 A          通道 B          通道 C          综合可靠性
Claude Code   Hooks ✅       CLAUDE.md ✅   扫 event.log ✅  ★★★★★
Kilo Code     MCP track ⚠️  AGENTS.md ✅   扫 event.log ✅  ★★★★☆
Hermes Agent  MCP track ⚠️  AGENTS.md ✅   扫 event.log ✅  ★★★★☆
OpenCode      — ❌          .opencode ✅   扫 event.log ✅  ★★★☆☆
OpenClaw      MCP track ⚠️  多文件 ✅      扫 event.log ✅  ★★★★☆
```

## 二、Claude Code（最完善）

### 连接方式

```json
// ~/.claude/settings.json
{
  "mcpServers": {
    "agent-achievements": {
      "command": "agpa",
      "args": ["mcp"]
    }
  }
}
```

MCP stdio 协议，LLM spawn + 管理生命周期。

### 事件采集

#### 通道 A：Hooks（核心）

CC 提供 29 个 Hook 事件类型。AGPA 使用其中 **10 个**来覆盖主要事件：

```json
// ~/.claude/settings.json
{
  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track session.start"
      }]
    }],
    "SessionEnd": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track session.end"
      }]
    }],
    "TaskCreated": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track task.start"
      }]
    }],
    "TaskCompleted": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track task.complete"
      }]
    }],
    "PreToolUse": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "agpa track tool.call"
      }]
    }],
    "PostToolUse": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "agpa track tool.complete"
      }]
    }],
    "PostToolUseFailure": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "agpa track tool.fail"
      }]
    }],
    "UserPromptSubmit": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track user.prompt"
      }]
    }],
    "PostCompact": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track context.compact"
      }]
    }]
  }
}
```

配置极简——`agpa track` 自动读取 `$CLAUDE_*` 环境变量补充参数。

Hook 触发时序：

```
SessionStart → session.start        ← 会话开始
      │
TaskCreated  → task.start           ← 用户下任务
      │
      ├── PreToolUse → tool.call    ← tool 开始执行
      │      │
      │      ▼
      ├── PostToolUse → tool.complete  ← tool 执行成功
      │  或
      ├── PostToolUseFailure → tool.fail ← tool 执行失败
      │      │
      │      ├── 继续当前 task → 又一轮 PreToolUse
      │      │
      │      └── 任务完成 → TaskCompleted → task.complete
      │
      ├── PostCompact → context.compact  ← 上下文压缩 (可选)
      │
      └── SessionEnd → session.end  ← 会话终止
```

**Hook 环境变量**（`agpa track` 自动读取，无需手动传参）：

```
$CLAUDE_TOOL_NAME         工具名（Bash, Read, Edit...）
$CLAUDE_TOOL_INPUT        tool call 参数（不进 payload，只记录存在）
$CLAUDE_TOOL_RESULT       tool 返回结果摘要
$CLAUDE_DURATION_MS       tool 执行耗时
$CLAUDE_CODE_SESSION_ID   session ID（匹配 hooks input 的 session_id）
$CLAUDE_SESSION_DURATION  session 持续时间
$CLAUDE_TASK_ID           task ID（TaskCreated/TaskCompleted 时）
```

#### 通道 A 补充：conversation.message

CC Hook 体系中没有对应 `conversation.message` 的 Hook。这个事件通过 MCP track 由 agent 主动上报，需要的 Hook 只有 `Stop`（触发 poll）：

```
In CLAUDE.md:
  在输出之前调用 achievement.track("conversation.message", {step_number, model})
```

#### 通道 B：CLAUDE.md 指令

```markdown
<!-- CLAUDE.md 追加内容 -->

<achievement_tracking>
- 在你输出每个对话轮次的回复前，调用 achievement.track("conversation.message", {"step_number": <当前对话轮次编号>, "model": "<当前模型名>"})
- 在 agent 收到用户新任务时，调用 achievement.track("task.start", {"task_id": <唯一任务编号>, "description_summary": "<任务简短描述>"})
- 在你完成某个用户请求并输出回复后，调用 achievement.track("task.complete", {"task_id": <编号>, "step_count": <此任务中执行的 step 数量>})
</achievement_tracking>

<achievement_display>
...
</achievement_display>
```

#### 可靠性评估

| 事件类型 | 采集方式 | 可靠性 | 备注 |
|---------|---------|--------|------|
| session.start | SessionStart A | ★★★★★ | Hook 100% 触发 |
| session.end | SessionEnd A | ★★★★★ | 带 reason 字段 |
| task.start | TaskCreated A | ★★★★★ | CC 内部 Task 系统事件 |
| task.complete | TaskCompleted A | ★★★★★ | 同上 |
| tool.call | PreToolUse A | ★★★★★ | 精确的 call-start 时间 |
| tool.complete | PostToolUse A | ★★★★★ | 自带 duration_ms |
| tool.fail | PostToolUseFailure A | ★★★★★ | 新增，之前无法捕获 |
| user.prompt | UserPromptSubmit A | ★★★★★ | 用户提交 prompt，user_partner/query_master |
| context.compact | PostCompact A | ★★★★★ | context_conscious 成就 |
| conversation.message | MCP B | ★★★★☆ | 唯一的非 Hook 事件，依赖指令遵守 |
| output.* / file.* | — | ☆☆☆☆☆ | 暂无捕获方式 |
| output.accept | 推断 | ★★☆☆☆ | 暂无精准捕获方式 |
| model.switch | MCP B | ★★★☆☆ | 依赖 agent 自觉上报 |

**综合可靠性**：★★★★★

## 三、Kilo Code

### 连接方式

Kilo Code 是 VS Code 扩展，MCP 配置在 VS Code settings 的 `kilocode.mcpServers` 段：

```json
// VS Code settings.json
{
  "kilocode.mcpServers": {
    "agent-achievements": {
      "command": "agpa",
      "args": ["mcp"]
    }
  }
}
```

**配置注入方式**：

```bash
agpa init --tool kilo-code
# → 写入 ~/.config/Code/User/settings.json 或 .vscode/settings.json
# → 优先写入 workspace 配置，fallback 到 user 配置
```

**检测逻辑**：

```
1. 尝试读 workspace 级 .vscode/settings.json
2. 如果不存在 → 读 user 级 settings.json
3. workspace 存在且已包含 MCP 配置 → 不重复
4. user 级存在且已包含 → 不重复
5. 两个都不存在 → 写入 user 级
```

### 事件采集

Kilo Code **没有 Hook 体系**，事件采集走双通道（MCP + 指令文件）：

#### 通道 A：MCP tool.call 自动录制

某些 MCP Server 实现会在 tool 被调用时自动记录。但取决于 Kilo Code 对 MCP 的具体实现，这个能力可能不保证。

**依赖**：Kilo Code 是否在 LLM 调用 MCP tool 前后注入追踪日志。

**实际评估**：Kilo Code 不做 tool.call 日志 → 通道 A 在此工具上降级为 ⚠️

#### 通道 B：AGENTS.md 指令（主通道）

```markdown
<!-- AGENTS.md 追加内容 -->

## Achievement Tracking

After each user message in a conversation:
1. Call achievement.track("conversation.message", {"step_number": <current_turn>, "model": "<unknown, don't know>", "tool_source": "kilo-code"})
2. If you just completed a user request/asked for follow-up, call achievement.track("task.complete", {"task_id": "<unique>", "step_count": <steps_taken>})
3. If you just started a new user request (after receiving the user's first message), call achievement.track("task.start", {"task_id": "<id>"})
4. At the end of your response, call achievement.poll().
5. If achievement.poll() returns pending achievements, display them before your main output using ANSI colored box.

## Tool Usage Tracking
After each tool use, call achievement.track("tool.complete", {
  "tool_name": "<name of the tool used>",
  "tool_result_summary": "<brief summary of what happened>",
  "duration_ms": <if you know>
})
```

#### 事件合成

agpa MCP Server 端可以对两个工具各自独立 track 的事件做去重：

```
Kilo Code 用户工作 2 小时：
  conversation.message × 8 条（MCP track B）
  tool.complete × 15 次（MCP track B）
  task.complete × 3 次（MCP track B）
  
→ event.log 汇总 → Evaluator 全量扫描
→ 成就条件对来源不敏感，只看计数
```

#### 可靠性评估

| 事件类型 | 采集方式 | 可靠性 | 备注 |
|---------|---------|--------|------|
| tool.complete | MCP B | ★★★★☆ | 依赖指令遵守，一般可靠 |
| conversation.message | MCP B | ★★★★☆ | 同上 |
| task.complete | MCP B | ★★★☆☆ | 依赖 agent 判断"任务完成"的节点 |
| session.end | — | ★★☆☆☆ | 无 Hook，无法捕获 |
| session.start | — | ★☆☆☆☆ | 无 Hook，无法捕获 |

**综合可靠性**：★★★★☆

## 四、Hermes Agent

### 连接方式

```yaml
# ~/.config/hermes/cli-config.yaml
mcp_servers:
  agent-achievements:
    command: agpa
    args: ["mcp"]
```

### 事件采集

Hermes Agent 与 Kilo Code 类似，没有自己的 Hook 体系，但有较好的指令遵守能力。

#### 通道 B：AGENTS.md 指令（主通道）

与 Kilo Code 的 AGENTS.md 指令内容相同。

**Hermes 对指令的遵守程度**：Hermes Agent 的架构设计上有更强的指令遵循能力（参考其 TOGETHER.md, 指令文件体系），所以同等级指令的实际效果可能比 Kilo Code 好。

#### 通道 A 补充：CLI wrapper

Hermes 是 CLI 工具（非 VS Code 扩展），可以 wrapper 方式增强事件采集：

```bash
# ~/.local/bin/hermes  → 替换为 wrapper 脚本
#!/bin/bash
# 调用原始 hermes
/usr/local/bin/hermes "$@" &
PID=$!
# 捕获 session.start
agpa track session.start --tool_source hermes
wait $PID
# 捕获 session.end
agpa track session.end --exit-code $?
```

但这种方式是**可选的增强**，不作为默认安装方案（因为需要替换用户已有的 alias/脚本）。

#### 可靠性评估

| 事件类型 | 采集方式 | 可靠性 | 备注 |
|---------|---------|--------|------|
| tool.complete | MCP B | ★★★★☆ | Hermes 指令遵守能力较强 |
| conversation.message | MCP B | ★★★★☆ | 同上 |
| task.complete | MCP B | ★★★☆☆ | 指令 + agent 判断 |
| session.end | wrapper (opt) | ★★★★☆ | 如果安装了 wrapper 脚本 |

**综合可靠性**：★★★★☆

## 五、OpenCode

### 连接方式

OpenCode 使用 TOML 配置文件：

```toml
# ~/.opencode/settings.toml
[mcp]
command = "agpa"
args = ["mcp"]
```

### 事件采集

OpenCode **没有 Hook**，是几个工具中事件采集能力最弱的。全部依赖指令文件。

#### 通道 B：.opencode/rules 指令

```markdown
<!-- .opencode/rules 追加内容 -->

## achievement tracking
after each turn: {
  call achievement.poll(),
  if pending: display achievements before output,

  call achievement.track("conversation.message")
}
after each tool call: {
  call achievement.track("tool.complete", {tool_name, result_summary})
}
after completing a user request: {
  call achievement.track("task.complete", {task_id, step_count})
}
```

OpenCode 的 `.opencode/rules` 相对其他工具的 AGENTS.md 来说，指令遵循程度较低。根据社区反馈，OC agent 有时会忽略 rules 中的非核心指令。

#### 降级策略

对于 OpenCode 用户，建议配合 `agpa init --tool opencode` 后，在 Dashboard 中着重提示"部分成就在 OpenCode 中无法解锁"：

```
⚠ OpenCode 对 events 的采集覆盖有限
  - ✅ conversation.message / tool.complete / task.complete
  - ⚠️ session.start / session.end / model.switch → 无法采集
  - 但最核心的 Onboarding + Milestone + Skill 类成就仍然有效
```

#### 可靠性评估

| 事件类型 | 采集方式 | 可靠性 | 备注 |
|---------|---------|--------|------|
| tool.complete | rules B | ★★★☆☆ | OC 指令遵守能力弱 |
| conversation.message | rules B | ★★★☆☆ | 同上 |
| task.complete | rules B | ★★☆☆☆ | 较难保证每轮都追踪 |
| session.start | — | ☆☆☆☆☆ | 无法采集 |
| session.end | — | ☆☆☆☆☆ | 无法采集 |

**综合可靠性**：★★★☆☆

```json
      "agent-achievements": {
        "transport": "stdio",
```

**配置注入方式**：

```bash
agpa init --tool openclaw

```

OpenClaw 有多文件 instructions 体系，事件采集比 Kilo/Hermes 更可控：

```bash
~/.openclaw/workspace/
├── AGENTS.md       ← 操作规则、安全约束、行为指令（poll + track 主通道）
├── TOOLS.md        ← 工具使用指南（tool 使用后 track）
```

### 事件采集三通道

#### 通道 A：MCP tool.call 自动录制

OpenClaw 对 MCP 的实现较为完善，agent 调用 tool 时 MCP Server 侧能感知到 tool.call 和 tool.complete。

**实际效果**：MCP Server 启动后，agent 每调用一个 tool，SDK 层面就能知道。但 tool 名称和参数是否能捕获取决于具体实现。

#### 通道 B：AGENTS.md 指令（poll 主通道）

```markdown
<!-- AGENTS.md 追加内容 -->

## Achievement Tracking
After each conversation turn, before you output your final response:
1. Call achievement.poll() via MCP
2. If pending achievements exist, display them before your response output
3. Call achievement.track("conversation.message", {
    "step_number": <current_turn>,
    "model": "<model_name if you know>",
    "tool_source": "openclaw"
  })
```

#### 通道 C：TOOLS.md 指令（track 补充）

```markdown
<!-- TOOLS.md 追加内容 -->

## Post-Tool-Use Tracking
After using any tool (Read, Edit, Bash, etc.):
1. Call achievement.track("tool.complete", {
    "tool_name": "<name>",
    "tool_result_summary": "<brief>",
    "tool_source": "openclaw"
  })
2. If this tool call completed a user's request, also call:
   achievement.track("task.complete", {
```

### 事件空白分析

**当前无法采集的事件**（任何工具都不支持）：

```
task.abandon    — 无法区分"放弃"和"中断"
output.*        — 无法知道用户是否接受了 agent 的输出（agent 不报告这个）
file.*          — 无法感知 agent 背后的文件操作
git.commit      — agent 可以做但不会主动报告
test.*          — 同文件操作，agent 跑测试但不会主动报告结果
```

**补偿策略**：

```
task.abandon:
  推断：如果 task.start 后超期没有 task.complete → 可能是 abandon
  不推荐，因为误报率高

output.accept/reject:
  目前不设计成就依赖这些事件
  未来如果 Agent SDK 支持用户反馈，可以补充

file.* / git.* / test.*:
  部分成就通过 tool.complete 的 payload 携带
  例如：如果 tool 是 Bash && command 包含 "git commit" → 可以认为是 git.commit 事件
  但这种"语义推断"可靠性较低，仅限特定成就使用
```

## 八、事件去重策略

### 重复场景

```
场景 1：同一事件被多个通道捕获
  tool.complete:
    CC:  PostToolUse Hook → agpa track tool.complete (通道 A)
    CLAUDE.md 指令 → agent 调用 MCP track tool.complete (通道 B)
  → 两个 agpa track 调用，event.log 中两条记录

场景 2：同一事件被自动录制 + 主动上报
  OpenClaw:
    MCP Server 自动记录 tool.call (通道 A)
    TOOLS.md 让 agent track tool.complete (通道 C)
  → 可能重复
```

### 去重算法

Evaluator 扫描 event.log 时做去重：

```python
def deduplicate(events: list[Event]) -> list[Event]:
    """基于 (event_type, session_id, timestamp_rounded) 去重"""
    seen = set()
    result = []
    for e in events:
        # 将 timestamp 向下取整到秒（同一个 session 里同一秒的同一事件类型 = 重复）
        key = (e.event_type, e.context.session_id, e.timestamp.rsplit('.')[0][:19])
        if key not in seen:
            seen.add(key)
            result.append(e)
    return result
```

**去重只影响计数类成就**（CounterEvaluator、ThresholdEvaluator）。序列类成就（SequenceEvaluator）和去重计数类（DistinctCountEvaluator）天然不受重复影响。

### 去重副作用

```
去重前:  tool.complete × 12 条（实际 tool 调用 8 次，4 条重复）
去重后:  tool.complete × 8 条
→ 计数类成就更准确

误杀: 同一秒内两次不同的 tool 调用
  tool.complete: { tool_name: "Read", timestamp: "10:00:01" }
  tool.complete: { tool_name: "Edit", timestamp: "10:00:01" }
  → 时间戳向下取整到秒后相同 → 可能误删一个

缓解: 使用 event_id（UUID）作为去重键，而不是 timestamp
  key = (event_type, context.session_id, event_id)  # 但 event_id 每次都不同
  → 不对，这等于不去重

正确方案:
  key = (event_type, context.session_id, payload_hash)
  payload_hash = sha256(sorted(payload))[:8]
  相同 event_type + 相同 session + 相同 payload → 视为重复
```

## 九、事件采集状态机

### 单次 session 的事件流

```
session.start  (CC: ✦, others: —)
      │
      ▼
task.start  (all: ★ / MCP track B)
      │
      ├── tool.call (CC: ✦, OC: ✦, others: —)
      │      │
      │      ▼
      ├── tool.complete (all: ★)
      │      │
      │      ├── 继续当前 task → 又一个 tool.call
      │      │
      │      └── 任务完成 → task.complete (all: ★)
      │
      ├── conversation.message (all: ★)
      │
      ├── model.switch (CC: ★, others: ✦)
      │
      ├── tool.deny (CC: ★, others: ✦)
      │
      └── 新 task 开始 → task.start (again)
      
session.end (CC: ★, others: —)
```

### 采集覆盖的成就影响

哪些成就受事件覆盖度影响：

```
不受影响（依赖的 event 全部工具都能采集）：
  first_contact      — conversation.message
  three_s_company    — conversation.message
  tool_time          — tool.complete
  happy_landing      — session.start（CC 才有）
  double_digits      — task.complete

  millennium         — task.complete
  file_whisperer     — file.edit/create（无工具能精确捕获 → 依赖推断）

受影响（依赖特定事件）：
  first_sunrise      — session.start（非 CC 用户无法精确捕获）

  weekend_warrior    — session.start（同上）

  对于 session.start：非 CC 工具用 conversation.message 首次出现的时间戳推断
  对于 model.switch：模糊匹配（event_type 变化时推断可能 model 变了）
```

### Session 推断算法

非 CC 工具无 session.start → 通过 conversation.message 推断：

```python
def infer_sessions(events: list[Event]) -> list[Session]:
    """从事件流推断 session 边界"""
    sessions = []
    current_session = None
    
    for e in sorted(events, key=lambda x: x.timestamp):
        if e.event_type == "conversation.message":
            if current_session is None:
                # 第一条 message → session start
                current_session = Session(start=e.timestamp)
            elif gap_exceeds_threshold(e.timestamp, current_session.last_time):
                # 消息间隔 > 30 分钟 → 新 session
                sessions.append(current_session)
                current_session = Session(start=e.timestamp)
            current_session.last_time = e.timestamp
        elif e.event_type == "session.end":
            if current_session:
                sessions.append(current_session)
                current_session = None
    
    if current_session:
        sessions.append(current_session)
    
    return sessions

GAP_THRESHOLD = 30 * 60  # 30 分钟无消息 → 视为新 session
```

这种推断对 `first_sunrise`（24h 内 ≥ 2 session）和 `night_owl`（0-5点有 session）等成就足够准确。

### v1.0 事件采集覆盖总结

```
可采集事件（10 类，覆盖主要成就需求）:
  session.start     — CC 精确 / 其余推断
  session.end       — CC 精确 / 其余不可用
  task.start        — 所有工具 MCP track
  task.complete     — 所有工具 MCP track
  tool.call         — CC Hook / OC MCP Server
  tool.complete     — 所有工具
  tool.deny         — CC Hook / 其余指令
  conversation.message — 所有工具 MCP track
  model.switch      — CC Hook / 其余指令

不可采集事件（当前版本不依赖）:
  task.abandon
  output.accept / output.edit / output.reject
  file.create / file.edit / file.delete
  git.commit / test.pass / test.fail
  session.start (非 CC)
  session.end (非 CC)
```

### v1.1+ 增强方向

```
1. Agent SDK 集成
   各工具如果提供 SDK/API，可以更精确捕获事件
   例：CC 的 Hook 环境变量扩展 → 添加 file.* 事件

2. 文件系统 Watch
   inotify/FSEvents 监听工作目录变化 → 捕获 file.* 事件
   可选增强，默认关闭（性能考虑）

   post-commit hook 自动 track git.commit
   用户手动配置 → agpa install-git-hook

4. Agent 日志解析
   如果 Agent 工具的日志格式公开，可以解析日志回放历史事件
   适合"我已经用 CC 一年了，现在想装 AGPA"的场景
```
