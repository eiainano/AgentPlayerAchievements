># 核心引擎设计：事件流 → 评估 → 展示

> 文档目的：定义 track() / EventBuffer / event.log / Evaluator / PendingQueue / poll() 的实现方案

---

## 一、数据流总览

```
track()  →  EventBuffer  →  event.log (JSONL)
                                │
                          (下次 poll 时)
                                │
                                ▼
                          Evaluator  →  state.json  →  PendingQueue
                                                      │
                                                poll() 返回
```

**核心设计决策：评估只在 poll() 时做，track() 只追加不评估。**
- `track()` 保证 < 1ms，无锁
- `poll()` < 5ms，加锁评估

## 二、track(event_type, payload)

### 功能

往 `event.log` 追加一行 JSON 事件记录。

### 事件数据结构

```json
{
  "protocol_version": "1.0",
  "event_id": "uuid",
  "timestamp": "ISO8601",
  "tool_source": "claude-code | kilo-code | opencode",
  "event_type": "task.complete | tool.call | session.start | ...",
  "payload": {},
  "context": {
    "session_id": "...",
    "model": "sonnet"
  }
}
```

### 算法

```
track(event_type, payload):
    1. 构造 event 对象（填充 timestamp / event_id / context）
    2. validate(event) — 检查 event_type 在标准化协议范围内
    3. line = json.dumps(event, sort_keys=True) + '\n'
    4. fd = open('event.log', O_APPEND | O_CREAT | O_WRONLY, 0644)
    5. fd.write(line)
       // POSIX 保证：单行 < PIPE_BUF (4096B) 时 O_APPEND 原子写入
    6. fd.close()
    7. return { status: "ok" }
```

## 三、EventBuffer

实际上不是内存缓冲区，而是三个职责：

### 职责 1：写入代理

```typescript
class EventBuffer {
  private fd: number;

  constructor(path: string) {
    // 'a' = O_APPEND | O_CREAT | O_WRONLY
    // LINE_BUFFERED: 遇到 \n 自动 flush
    this.fd = fs.openSync(path, "a");
  }

  append(event: Event): void {
    const line = JSON.stringify(event, Object.keys(event).sort()) + "\n";
    fs.writeSync(this.fd, line);
  }
}
```

**fsync 策略（三档可配）：**

| 策略 | 行为 | 可靠性 | 性能 |
|------|------|--------|------|
| 实时 | 每次 write 后 fsync() | ⭐⭐⭐⭐⭐ | 慢 |
| 定时（默认） | 每 5s fsync() | ⭐⭐⭐⭐ | 中 |
| 批量 | 每 100 条 fsync() | ⭐⭐⭐ | 快 |

### 职责 2：去重（可选）

同一事件可能被重复 track（如 agent 重试），用 `(event_type, payload_hash)` 去重。去重不放在 track 时，放在 Evaluator 扫描时做。

### 职责 3：水线游标

```
EventBuffer.watermark:
    记录 evaluator 上次读到了 event.log 的第几行
    存储在 state.json: { "last_evaluated_line": 4723 }

    目的：避免每次 poll() 都全量扫描所有历史行
```

## 四、event.log 文件格式

### JSON Lines（JSONL）

```
~/.agent-achievements/event.log
─────────────────────────────────
{"event_id":"a1","event_type":"conversation.message","timestamp":"...","tool_source":"claude-code",...}
{"event_id":"a2","event_type":"tool.complete","timestamp":"...",...}
{"event_id":"a3","event_type":"task.complete","timestamp":"...",...}
```

- 每行一个独立 JSON 对象
- 行尾 `\n` 分隔
- 无需逗号、无需外层数组
- `tail -f event.log` 即可实时查看

### 生命周期管理

```bash
event.log 越来越大时的处理方案：
  - 按月轮转: event-2026-05.jsonl, event-2026-06.jsonl
  - 归档压缩: gzip 旧文件
  - Evaluator 只扫描当前文件 + 水线
```

## 五、Evaluator（核心引擎）

### 调度时机

只在 `poll()` 被调用时触发。

```
poll():
    if PendingQueue 非空:         # 有待展示成就
        return 队首成就（最多 5 个）
    else:                         # 没有待展示成就
        evaluate()                # 执行评估
        return 新解锁成就
```

### evaluate() 主算法

```
evaluate():
    1. 获取文件锁 (flock LOCK_EX)
       // 防止两个 MCP Server 同时评估导致 state.json 竞争
    2. 读 watermark → event.log 从 watermark 往 EOF → events[]
    3. 更新 watermark
    4. 对每个未解锁的成就重新计算进度
       → 判断是否解锁
    5. 写回 state.json
    6. 释放锁 (flock LOCK_UN)
```

### 五种 Evaluator 实现

#### 1. CounterEvaluator（计数类）

适用：`first_contact`、`century`、`file_whisperer`、`streak_7`

```typescript
条件定义：
  { type: "counter", event: "task.complete", operator: ">=", value: 100 }

评估：
  const count = events.filter(e => e.event_type === "task.complete").length;
  progress.current = count;
  progress.target = 100;
  const unlocked = count >= 100;
```

按 event 类型计数，最简单。

#### 2. ThresholdEvaluator（阈值/窗口类）

适用：`first_sunrise`（24h 内 ≥ 2 session）、`short_circuit`（prompt ≤ 50 字符）

```typescript
条件定义：
  { type: "threshold", event: "session.start", window: "24h", operator: ">=", value: 2 }

评估：
  const now = Date.now();
  const relevant = events
    .filter(e => e.event_type === "session.start")
    .filter(e => (now - new Date(e.timestamp).getTime()) <= 24 * 3600 * 1000);
  const count = new Set(relevant.map(e => e.context.session_id)).size;
  const unlocked = count >= 2;
```

核心在 `window` 参数：只计算指定时间窗口内的事件。

#### 3. DistinctCountEvaluator（去重计数类）

适用：`mcp_explorer`（使用 5 个不同 MCP server）、`polyglot`（5 种编程语言）

```typescript
条件定义：
  { type: "distinct_count", event: "mcp.server_used", field: "server_name",
    operator: ">=", value: 5 }

评估：
  const distinct = new Set<string>();
  events
    .filter(e => e.event_type === "mcp.server_used")
    .forEach(e => distinct.add(e.payload.server_name));
  progress.current = distinct.size;
  const unlocked = distinct.size >= 5;
```

按指定字段去重计数。

#### 4. SequenceEvaluator（序列匹配类）

适用：`the_debugger`（报错 → 自修 → 通过）、`full_cycle`、`the_switch`

```typescript
条件定义：
  { type: "sequence", events: ["test.fail", "agent.self_fix", "test.pass"],
    window: "single_task" }

评估：
  按 task_id 分组 → 每组内按 timestamp 排序
  → 检查子序列匹配

  算法：
    let i = 0;
    for (const event of sortedEvents) {
      if (event.event_type === pattern[i]) {
        i++;
        if (i === pattern.length) {
          unlocked = true;
          break;
        }
      }
    }
```

允许中间穿插其他事件，核心是**有序子序列**匹配。

#### 5. StreakEvaluator（连续天计数类）

适用：`streak_7`、`streak_30`、`streak_100`

```typescript
条件定义：
  { type: "streak", event: "session.start", operator: ">=", value: 7 }

评估：
  取所有 session.start，按日期去重
  → 找到最长连续日期段

  // 例: [5/1, 5/2, 5/3, 5/5, 5/6, 5/7]
  // 最长连续 = 5/1-5/3 = 3天, 5/5-5/7 = 3天
  const current = 3;
  const unlocked = current >= 7;  // → false
```

## 六、PendingQueue

### 数据结构

`state.json` 中的 pending 段：

```json
{
  "version": 2,
  "last_evaluated_line": 4723,
  "pending": [
    {
      "id": "first_contact",
      "unlocked_at": "2026-05-17T14:30:00Z",
      "displayed": false
    }
  ],
  "achievements": {
    "first_contact": {
      "unlocked": true,
      "unlocked_at": "2026-05-17T14:30:00Z",
      "progress": { "current": 1, "target": 1 }
    }
  }
}
```

### poll() 消费 pending

```
poll():
    const pending = state.pending.filter(a => !a.displayed);
    const displayed = [...pending];
    for (const a of displayed) {
        a.displayed = true;
    }
    writeStateJson(state);
    return { pending: displayed, urgent: state.pending.length > 3 };
```

### 关键特性

- 成就在 poll() 返回后**不删除**，只标记 `displayed`
- 下次 poll() 不会再返回同一个成就
- `urgent: true`：堆积超过 3 个待展示，通知 agent 优先展示

## 七、并发安全

```
进程 A (CC)                     进程 B (OpenCode)
    │                              │
    ├── track() → event.log ←──────┤
    │   (无锁)         ↑            │
    │                  │ O_APPEND  │
    │                  │ 原子写入   │
    │                              │
    ├── poll()                     │
    │   ├── flock(LOCK_EX)         │
    │   ├── 读 event.log           │
    │   ├── 评估                   │
    │   ├── 写 state.json          │
    │   └── flock(LOCK_UN)         │
    │                              │
    │                     ├── poll()
    │                     ├── flock(LOCK_EX) ─── 排队等 A 释放
    │                     └── ...
```

### 两层安全设计

| 资源 | 锁策略 | 原因 |
|------|--------|------|
| **event.log** | 无锁，O_APPEND | 只追加不修改，POSIX 保证单行原子写入 |
| **state.json** | flock(LOCK_EX) | read-evaluate-write 需要互斥 |

### 最坏情况

```
两个进程同时 poll()：
  A 加锁 → 评估 5ms → 写 state → 解锁
  B 等待 → A 释放 → B 加锁 → 评估 → 写 state → 解锁

锁定总时间 ≈ 10ms，用户无感知
成就不会丢失（event.log 是完整记录）
```
