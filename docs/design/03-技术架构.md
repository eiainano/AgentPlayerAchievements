># 技术架构

## 一、核心设计

**一个 MCP Server，对接所有主流 Agent 工具。用户安装一句命令，之后零操作。**

```
                          ┌──────────────────────┐
                          │    agpa MCP Server      │
                          │                      │
 Claude Code ────┤        │  Tool: achievement.track(event)    │
 Kilo Code ──────┤        │  Tool: achievement.poll()          │
 Hermes Agent ───┤  MCP   │  Tool: achievement.stats()          │
 OpenCode ───────┤        │  Tool: achievement.showcase()      │
 OpenClaw ───────┤        │                      │
                          │  内部组件：            │
                          │  ├─ EventBuffer       │──── ~/.agent-achievements/
                          │  ├─ Evaluator         │      event.log
                          │  ├─ StateManager      │      state.json
                          │  └─ RarityEngine      │      ...
                          └──────────────────────┘
```

### 为什么是 MCP Server

```
1. 主流 Agent 工具全部支持 MCP
   Claude Code ✅  Kilo Code ✅  Hermes Agent ✅  OpenCode ✅  OpenClaw ✅

2. MCP Server 生命周期由 agent 自动管理
   Agent 启动 → spawn MCP server
   Agent 关闭 → 自动 kill
   用户不需要手动启动任何东西

3. 成就弹窗自然嵌入 agent 输出流
   agent 每轮结束调用 achievement.poll()
   → 有新成就 → agent 自己写进输出
   不是外部 hack，是 agent 行为的一部分

4. 跨工具无缝
   所有工具共享同一个 event.log + state.json
   成就累积是跨 tool 的
```

## 二、MCP Server 接口

```
Tool: achievement.track(event_type, payload)
  记录一个事件到 event.log（append-only，无锁）
  轻量，回调时间 < 1ms
  不评估成就条件（评估只在 poll() 时做）

Tool: achievement.poll()
  返回 pending 队列中未展示的成就
  格式：[{id, name, description, icon, rarity, set_progress}]
  pending 队列弹出，避免重复展示

Tool: achievement.stats()
  返回用户统计（总 task、总 tool call、等级、已解锁数等）

Tool: achievement.showcase(action, achievement_ids?)
  action: "view" | "set" | "auto"
  管理展示柜
```

## 三、成就展示：双通道渲染

成就解锁后通过两条独立通道展示：**终端内 ANSI 彩色弹窗**（主通道）+ **OS 原生通知**（副通道）。两条通道都在 `achievement.poll()` 的返回处理中触发，非外部注入。

### 3.1 渲染时序

```
Agent 每轮结束
        │
        ▼
achievement.poll()  ────── MCP Server
        │                      │
        │                   检查 pending 队列
        │                   有成就 → 返回 [{...}, ...]
        │                   无成就 → 返回 []
        ▼
agent 收到响应
        │
        ├── 通道 1 (终端内):  构造 ANSI 彩色框 → stdout
        │
        └── 通道 2 (OS 通知): 调用系统通知命令
```

### 3.2 通道 1：终端内 ANSI 彩色弹窗

Agent 根据 `achievement.poll()` 返回数据中的 `rarity` 字段选择配色，构造带 ANSI 转义码的输出。

**配色方案**（按稀有度）：

| 稀有度 | 边框色 | ANSI 256 | TrueColor | 名称色 | 描述色 |
|--------|--------|----------|-----------|--------|--------|
| Mythic | 玫红 | `196` | `#eb6f92` | 亮白粗体 | dim 细体 |
| Legendary | 金色 | `220` | `#ffd700` | 亮白粗体 | dim 细体 |
| Epic | 紫色 | `141` | `#c4a7fa` | 亮白粗体 | dim 细体 |
| Rare | 蓝色 | `39` | `#0087ff` | 亮白粗体 | dim 细体 |
| Uncommon | 绿色 | `78` | `#5fd7a7` | 亮白粗体 | dim 细体 |
| Common | 灰色 | `245` | `#8a8a8a` | 普通白 | dim 细体 |

### 3.3 通道 2：OS 原生通知

系统原生通知作为双保险——用户切到其他窗口时也能看到。

| 平台 | 命令 | 能力 |
|------|------|------|
| macOS | `osascript -e 'display notification ...'` | 标题 + 副标题 + 正文 + 声音 |
| Linux | `notify-send -i icon "title" "body"` | 标题 + 正文 + 图标 |
| Windows | PowerShell `Windows.UI.Notifications` | 标题 + 正文 + 图标 |

**macOS 示例**（弹窗效果）：
```
      ┌─────────────────────────────────┐
      │  🏆 Achievement Unlocked!       │
      │  The Debugger                   │  ← title
      │  Uncommon · Top 45%             │  ← subtitle

      │  它修好了。这就是信任。           │

```

**无声模式**：如果用户设置了 `achievements.notification.silent: true`，OS 通知跳过，仅在终端内展示。

### 3.4 `achievement.poll()` 返回格式

```json
{
  "pending": [
    {
      "id": "the_debugger",
      "name": "The Debugger",
      "description": "agent 代码报错，你让它自己修，修好了。这就是信任。",
      "icon": "🐛",
      "rarity": "uncommon",
      "rarity_percentile": 45.2,
      "rarity_color": { "border": "78", "hex": "#5fd7a7" },
      "xp": 100,
      "set": {
        "name": "The Bug Catcher",
        "progress": "1/2"
      },
      "unlocked_at": "2026-05-17T14:30:00Z",
      "urgent": false
    }
  ],
  "stats": {
    "total_unlocked": 23,
    "xp": 4720,
    "level": 7
  }
}
```

`rarity_color` 预计算好发给 agent，agent 不需要自己映射颜色。`urgent: true` 表示堆积超过阈值，agent 应优先展示。

### 3.5 兜底机制

如果 agent 某轮忘记调用 poll，成就堆积在 pending 队列中 ≤ 3 轮无妨。堆积超过阈值时，poll 返回 `urgent: true`。极少情况下成就延迟 1-2 轮展示，但不会丢失。

### 3.6 实测验证（2026-05-17）

**验证目标**：确认 ANSI 彩色输出能否在 Claude Code 中正常渲染。

**结论**：✅ 链路完全可行。

```
测试 1：Bash 工具输出
  python3 → stdout → Bash tool 抓取 → Claude Code 界面
  结果：❌ ANSI 转义码被转义，显示裸码

测试 2：Agent 直接输出
  agent 在回复中直接写入 ANSI 文本 → 终端模拟器
  结果：✅ 金色边框、绿色 Uncommon 标签、粗体名称全部正确渲染
```

**根因**：Claude Code 抓取工具 stdout 时对 ANSI 码做了转义处理，但 agent 自己的输出文本直通终端，不经过该层。

**适用性**：Claude Code 实测通过。Kilo Code、Hermes Agent、OpenClaw 均使用终端 stdout，同理可行。OpenCode 待验证。

## 四、事件采集

### 4.1 Claude Code（最完善）

CC 拥有 **29 种 Hook 事件类型**，足以精确捕获大部分标准事件。以下是与 AGPA 事件协议对应的 Hook 映射：

| 事件类型 | Hook 点 | 捕获方式 | 可靠性 |
|---------|---------|---------|--------|
| `session.start` | **SessionStart** | 会话创建/恢复时触发 | ★★★★★ |
| `session.end` | **SessionEnd** | 会话真正终止时触发，带 reason | ★★★★★ |
| `task.start` | **TaskCreated** | CC 内部 Task 系统创建 task | ★★★★★ |
| `task.complete` | **TaskCompleted** | CC 内部 Task 完成 | ★★★★★ |
| `tool.call` | **PreToolUse** | tool 执行前触发 | ★★★★★ |
| `tool.complete` | **PostToolUse** | tool 执行成功后，自带 duration_ms | ★★★★★ |
| `tool.fail` | **PostToolUseFailure** | tool 执行失败 | ★★★★★ |
| `conversation.message` | **Stop** + MCP track | Hook 触发 poll()，agent 通过 MCP 上报 | ★★★★☆ |
| `context.compact` | **PostCompact** | 上下文自动/手动压缩后 | ★★★★★ |
| `worktree.create` | **WorktreeCreate** | 创建工作树 | ★★★★★ |
| `worktree.remove` | **WorktreeRemove** | 移除工作树 | ★★★★★ |
| `model.switch` | MCP track | agent 自觉上报 | ★★★☆☆ |

**核心 Hook 配置**（`~/.claude/settings.json`）：

```json
{
  "mcpServers": {
    "agent-achievements": {
      "command": "agpa",
      "args": ["mcp"]
    }
  },
  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track session.start --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }],
    "SessionEnd": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track session.end --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }],
    "TaskCreated": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track task.start --task \"$CLAUDE_TASK_ID\" --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }],
    "TaskCompleted": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track task.complete --task \"$CLAUDE_TASK_ID\" --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }],
    "PreToolUse": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "agpa track tool.call --tool \"$CLAUDE_TOOL_NAME\" --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }],
    "PostToolUse": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "agpa track tool.complete --tool \"$CLAUDE_TOOL_NAME\" --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }],
    "PostToolUseFailure": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "agpa track tool.fail --tool \"$CLAUDE_TOOL_NAME\" --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }],
    "PostCompact": [{
      "hooks": [{
        "type": "command",
        "command": "agpa track context.compact --session \"$CLAUDE_CODE_SESSION_ID\""
      }]
    }]
  }
}
```

**优化选项：`type: "mcp_tool"`**

CC v2.1.118+ 支持 Hook 直接调用 MCP tool，无需经过 CLI 子进程。上述 `PostToolUse` 可改写为：

```json
{
  "type": "mcp_tool",
  "mcpServer": "agent-achievements",
  "tool": "track",
  "arguments": {
    "event_type": "tool.complete",
    "payload": {
      "tool_name": "$CLAUDE_TOOL_NAME"
    },
    "context": {
      "session_id": "$CLAUDE_CODE_SESSION_ID"
    }
  }
}
```

`mcp_tool` 方案减少一次进程 spawn，但 `command` 方案可移植性更好（同样可用于非 CC 场景的 cron/wrapper 调用）。**AGPA 默认使用 `command` 类型**，`mcp_tool` 作为可选优化。

所有 Hook 只做事件追加（< 1ms），不评估条件。评估在 `achievement.poll()` 被调用时做。

### 4.2 Kilo Code / Hermes Agent / OpenCode / OpenClaw

这四个都支持 MCP，但没有 CC 那样完善的 Hook 体系。事件采集走双通道（OpenClaw 走三通道，详见 4.3）：

```
通道 1（主）：MCP tool 调用本身即事件
  agent 每次调用 tool → 就是一次 tool.call 事件
  MCP server 在 tool 被调用时自动记录

通道 2（补）：instructions 文件指令
  AGENTS.md / .opencode/rules 中指令：
  "After each tool use, call achievement.track with the tool name and result."
  依赖 agent 遵守指令（同 poll 逻辑）
```

**关键设计**：即使通道 2 偶尔遗漏，也不影响核心成就（大部分成就基于累计统计和 poll 时的事件回放，而非实时采集每个事件）。累积成就不怕丢事件，只有连续类成就（streak）对完整性敏感。

### 4.3 OpenClaw（MCP + 多文件指令注入）

OpenClaw 原生支持 MCP，且有丰富的 instructions 文件体系：

```
~/.openclaw/openclaw.json  ← MCP server 配置
~/.openclaw/workspace/     ← Instructions 文件目录
  ├── AGENTS.md            ← 操作规则、安全约束、行为指令
  ├── SOUL.md              ← 人格、语调、边界定义
  ├── TOOLS.md             ← 工具使用指南
  ├── USER.md              ← 用户偏好
  ├── IDENTITY.md          ← Agent 身份
  ├── HEARTBEAT.md         ← 心跳/状态
  └── MEMORY.md            ← 记忆
```

**MCP 配置方式**：

```bash
# 注册 MCP server（写入 openclaw.json）
openclaw mcp set agent-achievements '{
  "command": "aa",
  "args": ["mcp"]
}'
```

`openclaw mcp` 子命令直接管理 `mcp.servers` 段，支持 stdio / SSE / Streamable HTTP 三种传输。

**事件采集**：

```
通道 1（主）：MCP tool 调用本身即事件
  agent 每次调用 tool → 自动生成 tool.call 事件
  MCP server 在 tool 被调用时记录

通道 2（补）：AGENTS.md 指令
  在 AGENTS.md 中注入：
  "After each conversation turn, call achievement.poll. 
   If achievements returned, display them before main output."
  
通道 3（补）：TOOLS.md 指令
  在 TOOLS.md 中注入：
  "After each tool use, call achievement.track with tool name and result."
```

**安装**：`agpa init --tool openclaw` → 调用 `openclaw mcp set` 注册 MCP server + 注入 AGENTS.md / TOOLS.md。

**可靠性**：MCP 注册由 config 文件保证，指令遵守依赖 agent 行为。综合评级 ★★★★☆。

## 五、安装：`agpa init`

用户感知只有一行命令：

```bash
agpa init --tool claude-code
agpa init --tool kilo-code
agpa init --tool hermes
agpa init --tool opencode  
agpa init --tool openclaw
```

每个 `agpa init` 自动完成：

```
1. 注册 MCP server 配置
   CC:       settings.json 的 mcpServers 段
   Kilo:     VS Code settings 的 kilocode.mcpServers
   Hermes:   cli-config.yaml 的 mcp_servers
   OC:       opencode.json / TOML 的 mcp 段
   OpenClaw: openclaw.json 的 mcp.servers 段 （openclaw mcp set）

2. 配置事件采集
   CC:       settings.json 的 hooks 段
   其他:     依赖 instructions 文件中的指令
   
3. 注入 instructions 文件
   CC:       CLAUDE.md 追加 achievement poll 指令
   Kilo:     AGENTS.md 或 system instructions
   Hermes:   AGENTS.md
   OC:       .opencode/rules
   OpenClaw: AGENTS.md + TOOLS.md

4. 初始化本地状态
   ~/.agent-achievements/ 目录 + 默认文件
```

之后每次启动 agent，MCP server 自动拉起。用户零操作。

### aa 进程模型

```
agpa 是一个单体小二进制，只干四件事：

  agpa init     写配置 + 注入指令（一次性操作，不是 daemon）
  agpa mcp      作为 MCP Server 被 agent spawn，提供 5 个 tool
  agpa track    CLI 模式追加事件（给 CC Hook 用）

不联网、不监听端口、不维持长连接。

生命周期完全由 agent 管理：
  CC 启动  → spawn agpa mcp (PID A)
  CC 关闭  → kill PID A

  同时开 CC + Hermes：
  CC     ── spawn agpa mcp (PID A) ──┐
                                    ├── 共享 ~/.agent-achievements/
  Hermes ── spawn agpa mcp (PID B) ──┘

99% 的调用只做极轻量操作：
  achievement.track()  → append 一行 JSONL               < 1ms
  achievement.poll()   → 扫 pending 队列，通常返回 []     < 2ms
  仅当条件满足时才做评估 → flock state.json → 入 pending  < 5ms
```



**TypeScript**。2026-05-17 确认。理由：
- Claude Code 生态的 MCP servers、plugins、skills 90%+ 用 TS/JS
- 用户习惯 `npx` / `npm install`，贴合生态默认
- MCP SDK (`@modelcontextprotocol/sdk`) 原生 TypeScript
- 无需编译、交叉编译，用户自构建门槛最低
- 全部核心机制已验证可行：MCP stdio ✅ | event.log O_APPEND ✅ | flock ✅ | 4 Evaluator ✅ | CLI ✅ | 跨平台 ✅ | ANSI ✅ | OS 通知 ✅ | 文件操作 < 2ms ✅

## 十、跨 Tool 累积

同一用户的成就跨工具共享。上午 CC，下午 Kilo Code，成就进度自动合并：

```
Profile 展示：
  "Century: 100 tasks"
  "来源: 72 Claude Code, 28 Kilo Code"
```

### 统一统计机制

event.log 中每条记录带 `tool_source` 字段，但**成就评估不关心来源**。评估器扫描 event.log 时做的是全量聚合：

```
用户一天内：
  CC 新对话 6 条  ──┐
  OpenCode 新对话 4 条 ─┼── Evaluator 扫描 event.log
                         │   → counter(event=conversation.message) >= 10
                         │   → ✅ "Three's Company" 计数到 10！
                         │   → 不管来源，只管总数
  Kilo Code 在忙别的 ───┘
```

tool_source 仅用于 Profile 展示（"来源: 72 CC, 28 Kilo"），不参与条件评估。

### 并发写入安全

两个工具同时运行时，各自的 MCP server 进程同时向 `~/.agent-achievements/event.log` 追加事件。JSONL 格式天然支持并发追加：

```
MCP 进程 A (CC):    ─── write("{\"tool_source\":\"claude-code\",...}\n") ───┐
MCP 进程 B (OpenCode): ─── write("{\"tool_source\":\"opencode\",...}\n") ───┤  O_APPEND 保证原子追加
                                                                          │
                                                                          ▼
                                                                  event.log 安全合并
```

- 每个事件 = 一行 JSON（< 512 bytes），`write()` 系统调用单行原子写入
- 文件以 `O_APPEND` 模式打开，POSIX 保证小写入（< PIPE_BUF）的原子性
- 评估器在 `achievement.poll()` 被调用时扫描 event.log，不关心哪行是谁写的

### state.json 并发安全

event.log 只追加不冲突，但 state.json 存在 read-evaluate-write 竞态：

```
agpa (CC)   读 state.json → 评估条件 → 写 state.json
                                           ↑
agpa (OC)   读 state.json → 评估条件 → 写 state.json
                                           ↑
                       后写的可能覆盖先写的解锁记录
```

最坏情况：成就重复弹出，或进度短暂回退。成就不会丢失（event.log 是完整记录）。解决：`poll()` 评估阶段加文件锁。

```
agpa (CC)   flock(LOCK_EX) → 读 state.json → 评估 → 写 state.json → flock(LOCK_UN)
agpa (OC)   flock(LOCK_EX) ──── 排队等待 ────→ 读 → 评估 → 写 → flock(LOCK_UN)
```

评估耗时 < 5ms，排队几乎无感。`track()` 只追加 event.log 不加锁，不阻塞任何一方。

### 可选 Profile 隔离

如果用户希望分工具统计（如 CC 和 OpenCode 各有独立成就进度），可通过 profile 机制：

```bash
agpa init --tool claude-code --profile work
agpa init --tool opencode  --profile personal
```

不同 profile 对应独立的 state.json 和 event.log，成就进度完全隔离。默认不使用 profile flag 时，所有工具共享同一套数据。

### 视觉设计

- **整体气质**：开发工具的信息架构（清晰、高效扫描）+ 游戏化视觉细节（发光边框、渐变、hover 动画、稀有度配色）。平时简洁克制，细节处在交互时展露
- **稀有度配色**：与终端 ANSI 一致 —— Mythic 玫红 / Legendary 金 / Epic 紫 / Rare 蓝 / Uncommon 绿 / Common 灰
- **Level 环**：conic-gradient 渐变色环（金→紫→金），中央 Level 数字（渐变白→紫）。Level 环颜色随等级变化：前 10 级灰色→50 级金色→100 级彩虹脉冲
- **XP 进度条**：渐变色填充（绿→金）。解锁新成就的瞬间进度条尾端播放粒子消散动画（CSS `@keyframes`，~500ms）
- **卡片 hover**：上浮 2-3px + 边框增亮 + Legendary 卡片有金色顶线发光 + 微弱 box-shadow
- **卡片特有动效**：Legendary 成就卡片常态带微弱呼吸光效（`animation: pulse 3s ease-in-out infinite`）；hover 时稀有度顶线发光加深
- **锁定成就**：opacity 0.35，hover 回升到 0.55
- **暗色主题**：固定暗色 `#0c0c14` 底色，暂不做 light mode
- **毛玻璃导航**：sticky 顶栏，`backdrop-filter: blur`，半透底
- **响应式**：手机可看，4 列 grid 在小屏缩为 2 列或 1 列

### 解锁动画与装饰奖励

成就解锁时在 Dashboard 内触发的视觉反馈（与终端 ANSI 弹窗独立）：

- **Toast 动画**：页面右下角弹出横幅（3 秒自动消失），显示成就名 + 稀有度色 + icon。与终端弹窗保持相同稀有度配色
- **Mythic 成就解锁**：触发全屏金色粒子爆发动画（CSS `@keyframes`，~2s），覆盖全屏后自然消散。仅首次解锁时触发
- **套装集齐奖励**：每个套装完成时触发对应视觉效果（Profile 背景更换、边框变色等），YAML 中 `reward.type` 控制：
  - `title` — 称号显示在 Profile 用户名下方
  - `background` — Profile 页背景纹理切换（9 种套装各一种）
  - `badge` — 展示柜底部绶带
  - `animation` — 解锁动画加速等行为效果
- 所有动画纯 CSS 实现（`@keyframes` + `transition`），不涉及额外 JS 库

### 技术实现

- 前端：纯静态 HTML + CSS + 原生 JS（或轻量框架如 Preact），打包进 npm 包
- 后端：`agpa dashboard` 起 Node.js HTTP server，仅 `GET` 路由：
  - `/` — 返回 SPA 入口
  - `/api/achievements` — state.json + achievements.yaml 合并后 JSON
  - `/api/stats` — stats.json
  - `/api/timeline` — history.json
- 只绑 `127.0.0.1:3867`，不暴露网络
- 数据只读，Dashboard 不做任何写入

## 十三、全球统计 & 稀有度（Opt-in）

用户可选择加入全球统计，从而在成就详情中看到"XX% 的人获得了此成就"。默认关闭。

### Opt-in 机制

```bash
agpa stats opt-in              # 加入全球统计
agpa stats opt-out             # 退出（已上传数据标记删除）
```

首次 `agpa init` 时询问，或之后在 Dashboard 中一键开启。**不默认勾选**，用户主动操作才生效。

### 上传数据

上报内容极简——**只有成就 ID 列表 + 匿名用户指纹**：

```json
{
  "user_hash": "sha256(uuid + salt)",   // 不可反推用户身份
  "achievements": {
    "first_contact":  { "unlocked": true,  "unlocked_at": "2026-01-15" },
    "century":         { "unlocked": false, "progress": { "current": 67, "target": 100 } },
    "the_debugger":    { "unlocked": true,  "unlocked_at": "2026-03-02" },
    "millennium":      { "unlocked": false, "progress": { "current": 142, "target": 1000 } }
  },
  "level": { "xp": 4720, "level": 7 },
  "tool_sources": ["claude-code", "opencode"],
  "updated_at": "2026-05-17T..."
}
```

**绝不上传**：prompt 内容、代码、文件路径、仓库名、IP 地址、用户名、邮箱。

### 服务端聚合

服务端只管计数，不存储个体可追踪数据：

```
收到 10,000 份上报 →
  每个成就 → unlocked_count / total_count = 稀有度百分比
  返回 → { "the_debugger": { "unlock_rate": 45.2, "tier": "uncommon" } }
```

用户 hash 只用于去重（同一用户多次上报覆盖旧数据）。

### 数据流架构

不收数据时不联网，无需维持 24×7 服务器进程。唯一的"后端"是一个云函数 + 一个静态文件。

```
 收数据（轻量云函数）            发数据（静态文件 + CDN）
 ════════════════              ════════════════════

 用户 opt-in ──POST──→ SCF 云函数     agpa mcp 启动
                       │                │
                       ├─→ 对象存储      ├─ 检查本地缓存时间
                       │   (COS/R2)     ├─ > 24h → GET rarity.json
                       │                │            (CDN 静态文件)
                       │                ├─ < 24h → 用本地缓存
                       │                └─ 网络不通 → 缓存兜底
                       │
                   CI 定时拉存储数据
                   算出每个成就的持有率
                   生成 rarity.json
                   写回对象存储 + CDN 刷新
```

用户日常使用只做一次 GET 拉静态 JSON（每天最多一次），不上传任何数据。只有 opt-in 用户首次上报和后续每日同步时发一次 POST。

### 云平台无关

不绑特定云。本质只需要两个东西：HTTP POST 入口 + 静态文件托管。

| 功能 | 腾讯云 | Cloudflare | AWS | 阿里云 |
|------|--------|-----------|-----|--------|
| 收数据 (POST) | SCF 云函数 | Workers | Lambda | 函数计算 |
| 存数据 | COS | R2/KV | S3 | OSS |
| 发 rarity.json | CDN 回源 COS | Pages/R2 | CloudFront+S3 | CDN+OSS |

任选一个即可，服务端代码开源，用户也可自部署。

### 成本：几乎免费

| 规模 | 腾讯云月费 | Cloudflare 月费 | 关键瓶颈 |
|------|-----------|----------------|---------|
| 1,000 用户 | **¥0** | **¥0** | 远低于免费额度 |
| 10,000 用户 | **¥0** | **¥0** | 免费额度内 |
| 100,000 用户 | **~¥3** | **¥0** | SCF 超出免费 200 万次=¥2.66 |
| 1,000,000 用户 | **~¥30** | **~$5** | 开始轻微收费 |

100 万用户规模之前基本不用掏钱。一人开发者，纯零成本起步。

### 稀有度更新时机

```bash
# 用户端自动更新（无需手动操作）
agpa mcp 启动时:
  1. 读 ~/.agent-achievements/rarity_cache.json 的 updated_at
  2. 距上次 > 24h → GET https://stats.agpa.dev/rarity.json
  3. 写入缓存 → 本次及后续查询使用最新数据

# 服务端更新（CI 自动）
GitHub Actions cron (daily):
  1. 从对象存储拉全量用户数据
  2. 逐个成就算持有率
  3. 分档: Common(>50%) / Uncommon(20-50%) / Rare(5-20%) / Epic(1-5%) / Legendary(0.1-1%) / Mythic(<0.1%)
  4. 生成 rarity.json → 写回对象存储
```

Dashboard 中提供「立即刷新」按钮，手动强制拉取。

### 用户端展示

```
Rarity: Uncommon · Top 45.2%     ← 动态，来自全球统计
Rarity: Uncommon · Top ~45%      ← 离线，来自 rarity_cache.json 缓存值
```

Dashboard 中额外展示：
- "你比 78% 的用户拥有更多成就"
- "你的 The Debugger 在它只有 2% 的时候就拿到的"（早期解锁荣誉）

### 隐私保证

- Opt-in 关闭时，零网络请求
- 用户可随时删除已上传数据（服务端按 user_hash 标记删除）
- 不计入 DAU/MAU，不关联第三方广告
- 服务端代码开源，可自部署
